# Доступ к дашборду из любой точки (4G) и всегда

## Рекомендуемая схема

- Если у вас уже есть Cloudflare-домен и прокси, лучший прод-вариант: `Cloudflare Pages`.
- Исходники живут в GitHub, а `Cloudflare Pages` собирает и отдает статическую версию дашборда постоянно.
- `Cloudflare Access` закрывает дашборд только для вас или узкого списка email.
- Локальный сервер на ПК остается только для интерактивного режима и локальной записи в `idea_inbox`.

## Почему это лучший вариант

- Не зависит от того, включен ли ваш домашний ПК.
- Работает с телефона по 4G как обычный публичный сайт.
- Не требует держать открытые порты дома.
- Уже совместим с текущей архитектурой дашборда: основной UI читает статические `data/*.json`.

## Что уже готово в проекте

- Автодеплой в `GitHub Pages`: `.github/workflows/deploy-pages.yml`
- Публичная сборка с санитизацией локальных путей: `scripts/dashboard/build_public_site.mjs`
- Публичный fallback для карты смыслов: если `/api/idea-inbox` нет, новые идеи сохраняются как локальные черновики в браузере устройства.
- Пошаговая инструкция для Cloudflare: `docs/cloudflare_pages_setup.md`

## Как запускать процесс обновления

1. Обновить данные локально: `.\scripts\dashboard\sync_workspace_data.ps1`
2. Закоммитить изменения в репозиторий.
3. Запушить в `main` или `master`.
4. Дождаться выполнения workflow `Deploy Dashboard to GitHub Pages`.

## Роль прокси

- Если нужен только постоянный доступ, `GitHub Pages` уже достаточно.
- Если нужен свой домен и проксирование, прокси должен смотреть на `GitHub Pages`, а не на локальный ПК.
- Если прокси смотрит на домашний сервер, это уже не режим "всегда": он ломается при выключенном ПК, смене IP или сбое домашней сети.

## Ограничение

- Общая запись идей с телефона в единый `idea_inbox` без локального сервера пока не реализована.
- На публичном сайте идеи сохраняются только как локальные черновики браузера.
